SYSTEM DESIGN PROMPT -- VALIDATION & VERIFICATION ARCHITECTURE
Context
We are building a fintech application for gig workers where users complete a multi-step onboarding process. Each step collects specific data and must pass a structured validation and verification workflow.
The system follows a hybrid architecture:
* Verification Layer -> Server side (FastAPI + MongoDB simulation of government APIs)
* Validation Layer -> On-device processing (inside the mobile application)
All steps must follow the same validation structure.
Validation Structure (Mandatory for Every Step)
1. Individual Validation
Each input field is validated independently on the user device.
2. Cross Internal Validation
Data within the same step is compared for consistency.
3. Cross Step Validation (Global Validation)
Data from the current step is compared with previously verified data stored in a global profile object.
4. Verification (Server)
If the step requires external verification (example: PAN, Aadhaar, Bank), the app calls a backend API that simulates government verification services.
After a step passes all validations, the system updates the global verification state and allows the user to proceed.

STEP-1 -- BASIC PROFILE INFORMATION (UPDATED)
Purpose
Collect the user's basic identity and work category.
This information initializes the global profile and determines which workflow will appear in Step-5.
Inputs
User must provide the following fields:
* Full Name
* Age
* Mobile Number
* Current Address
* Permanent Address
* Work Type
* Do you have a vehicle? (Yes / No)
Work Type Options
Platform Worker
Examples: Swiggy, Rapido, Uber, Zomato
Vendor / Seller
Examples: Street vendor, vegetable seller, shop owner
Skilled Tradesperson
Examples: Electrician, plumber, carpenter, painter
Freelancer
Examples: Designer, developer, photographer
The selected work type dynamically determines the workflow for Step-5.
Vehicle Ownership Field
User must answer the question:
Do you have a vehicle?
Options:
Yes
No
If the user selects Yes, Step-7 will require Vehicle Insurance upload.
If the user selects No, Vehicle Insurance becomes optional in Step-7.
Life Insurance and Health Insurance remain optional in Step-7 regardless of vehicle ownership.

SYSTEM DESIGN PROMPT -- STEP-2 IDENTITY (KYC) VERIFICATION


Context
Step-2 performs identity verification using Aadhaar and PAN. This step ensures that the documents belong to the same user by combining server-side verification APIs with on-device OCR validation and face matching.


The application uses simulation APIs that mimic government verification systems.


Verification APIs used in this step:


* Aadhaar Verification API (UIDAI Simulation)
* PAN Verification API (Income Tax Department Simulation)


These APIs are implemented in the backend using:


FastAPI + MongoDB mock databases


---


INPUTS COLLECTED


User must provide the following inputs:


* Aadhaar Number
* Aadhaar Card Photo
* PAN Number
* PAN Card Photo
* Live Selfie Photo

SYSTEM DESIGN PROMPT -- STEP-3 BANK ACCOUNT VERIFICATION
Context
Step-3 verifies the user's financial identity using bank account details and bank statements. The system combines server-side verification APIs with on-device OCR validation.
Verification APIs used in this step are simulated services implemented using FastAPI + MongoDB.
Separate MongoDB collections are used for each verification system:
* ifsc_db -> IFSC verification
* bank_accounts_db -> bank account verification (penny drop simulation)
The goal of this step is to confirm that the bank account belongs to the user and that the user has consistent transaction history.


INPUTS COLLECTED
User must provide the following bank details:
* Bank Name
* Account Holder Name
* Bank Branch Name
* IFSC Code verfify
* MICR Code
* Account Number
After bank verification:
* Upload Bank Statement (PDF or image)
* Optional: Upload UPI / GPay statement
User can provide maximum two bank accounts:
Primary Bank Account (required)
Secondary Bank Account (optional)
No more than two accounts are allowed.

Statement Period:
From: 01-Aug-2025
To: 31-Jan-2026


STATEMENT PERIOD VALIDATION
Using the OCR-extracted fields, the system validates that the statement covers the most recent 6 months.
Validation rules:
duration = to_date − from_date
Condition 1
duration >= 6 months
Condition 2
current_date − to_date <= 30 days
Example valid period
From: 01-Aug-2025
To: 31-Jan-2026
Example invalid period
From: 01-Nov-2025
To: 31-Jan-2026
Only if these conditions pass is the statement accepted.

SYSTEM DESIGN PROMPT -- STEP-4 UTILITY BILL VERIFICATION
Context
Step-4 evaluates the user's financial stability by verifying recurring utility payments. The system collects utility bills, performs OCR extraction, verifies some utilities using simulated government APIs, and validates payment consistency using bank statements from Step-3.
Validation in this step follows three layers:
Individual Validation (each bill)
Cross-Internal Validation (across 6 bills of the same utility)
Global Validation (common consistency across utilities)


UTILITIES COLLECTED
Mandatory utilities
* Electricity Bills (EB)
* Gas / LPG Bills
* Mobile Recharge Bills
* Rent (placeholder)
Optional utilities (score boost)
* WiFi / Broadband Bills
* OTT Subscription Bills
Each utility requires 6 latest bills.


ELECTRICITY BILL VERIFICATION
User uploads 6 electricity bills.
Verification API (simulation)
POST /api/gov/eb/verify
Request
{
"service_number": "123456789"
}
Example API response
{
"valid": true,
"service_number": "123456789",
"connection_status": "Active"
}


OCR EXTRACTION (EB)
OCR extracts:
Service Number
Bill Date
Payment Date
Bill Amount


INDIVIDUAL VALIDATION (EB)
For each EB bill:
* Service number extracted by OCR must match API service number
* Connection status must be Active
Consumer name is NOT validated because gig workers may live in rented houses and the EB account may belong to the landlord.


BANK TRANSACTION MATCHING (EB)
Payment is verified by matching with bank statement transactions.
Matching rule
transaction_date ≈ bill_payment_date
transaction_amount ≈ bill_amount


CROSS-INTERNAL VALIDATION (EB)
After all 6 bills are processed:
Check that all bills belong to the same electricity connection.
Rule
Service Number must be identical across all 6 bills.
If consistent -> Electricity utility verified.

BANK TRANSACTION MATCHING (LPG)
LPG refills may sometimes be paid in cash.
Therefore bank transaction matching is optional, not mandatory.
If a matching transaction is found -> additional verification signal.
If no transaction exists -> bill can still be valid.


CROSS-INTERNAL VALIDATION (LPG)
After processing all 6 LPG invoices:
Consumer number must remain identical across all invoices.
If consistent -> LPG utility verified.


MOBILE RECHARGE BILL VALIDATION
Users upload 6 recharge invoices from telecom apps.

Recharge payments are validated by matching with bank statements.
Matching rule

merchant contains telecom provider name


CROSS-INTERNAL VALIDATION (MOBILE)
After all 6 invoices are processed:
Mobile number must remain identical across all invoices.
If consistent -> Mobile utility verified.


WIFI / BROADBAND BILLS (OPTIONAL)
User uploads 6 broadband invoices.
OCR extracts
Account ID
Customer Name
Invoice Date
Amount
Validation
* Account ID must be consistent across all invoices
* Payment must match bank statement transactions.


OTT SUBSCRIPTION BILLS (OPTIONAL)
Examples
* Netflix
* Amazon Prime
* Disney+ Hotstar
OCR extracts
Subscription platform
Payment date
Amount
Validation
* Bills must represent monthly subscriptions
* Payment must match bank transaction.


RENT MODULE
Rent validation module is currently under development.
Rent = placeholder for future implementation.


GLOBAL VALIDATION (STEP-4)
Global validation is performed only after all 6 bills of each utility pass cross-internal validation.
The system checks common consistency signals across utilities.
Common fields evaluated:
* Utility account identifiers (service number, consumer number, mobile number)
* Payment frequency consistency
* Payment timestamps across months
If mandatory utilities are verified
Electricity + Gas + Mobile + Rent
then Step-4 is considered successful.
Optional utilities (WiFi, OTT) increase financial stability signals.

SYSTEM DESIGN PROMPT -- STEP-8 INCOME TAX RETURN (ITR) VERIFICATION


Context
Step-8 verifies the user's Income Tax Return (ITR) to establish declared annual income and confirm tax filing behaviour.


ITR is filed with the Income Tax Department of India and is linked to the user's Permanent Account Number (PAN).


This step helps evaluate:


* Income consistency
* Financial credibility
* Government-declared income records


Validation architecture follows:


OCR Extraction (on-device)
Individual Validation (field format validation)
Cross-Internal Validation (within ITR fields)
Cross-Step Validation (with PAN and identity data)
Server Verification (ITR acknowledgement verification)


All inputs in this step are optional.
If a user uploads an ITR document, the system must validate it.


---


INPUTS


User may upload:


* ITR Acknowledgement Receipt (PDF or Image)


---


OCR EXTRACTION


When the ITR acknowledgement document is uploaded, OCR extracts the following fields.


Field | Description
taxpayer_name | Name of taxpayer
pan_number | Permanent Account Number
assessment_year | Financial assessment year
acknowledgement_number | Unique ITR filing acknowledgement number
date_of_filing | Date when return was filed
total_income | Total income declared in return
gross_total_income | Income before deductions
deductions | Total deductions claimed
net_taxable_income | Final taxable income


Example OCR Output


{
"taxpayer_name": "Ravi Kumar",
"pan_number": "ABCDE1234F",
"assessment_year": "2024-25",
"acknowledgement_number": "123456789012345",
"date_of_filing": "2025-07-18",
"total_income": "480000"
}


---


VERIFICATION FIELDS


The following fields are used for backend ITR verification:


* PAN Number
* Acknowledgement Number
* Assessment Year


---


INDIVIDUAL VALIDATION (ON DEVICE)


Each extracted field is validated independently.


---


Taxpayer Name


Rules:


* Must contain alphabet characters
* Must contain minimum two words
* Allowed characters: alphabets, spaces, dots


Example valid


Ravi Kumar
R. K. Kumar


---


PAN Number


Rules:


* Must follow PAN format


Validation regex


[A-Z]{5}[0-9]{4}[A-Z]


Example valid


ABCDE1234F


Validation checks


* exactly 10 characters
* first 5 letters
* next 4 digits
* last 1 letter


---


Assessment Year


Rules:


* Must follow financial year format


Example


2024-25


Validation checks


* valid financial year format
* cannot be future year


Accepted range


Last three assessment years


Example


2022-23
2023-24
2024-25


---


Acknowledgement Number


Rules:


* Must be numeric
* Must contain exactly 15 digits


Example


123456789012345


---


Total Income Declared


Rules:


* Must be numeric
* Must be greater than zero


Example


480000


---


CROSS-INTERNAL VALIDATION


The system checks internal consistency between extracted ITR fields.


---


Name Consistency


Taxpayer name must remain consistent throughout the document.


Matching rule


Exact match OR fuzzy match >= 85%


---


Assessment Year and Filing Date Validation


The filing date must correspond to the assessment year.


Example


Assessment Year = 2024-25
Filing Date must occur during the valid filing period.


---


CROSS STEP VALIDATION


The system compares ITR information with previously verified profile data.


---


PAN Cross Check


ITR PAN must match:


* PAN verified in Step-2


Example


ITR PAN ↔ PAN from PAN verification step


---


Name Cross Check


ITR taxpayer name must match:


* name verified from PAN in Step-2
* name provided in Step-1 profile


Matching logic


Exact match OR fuzzy match >= 85%


---


INCOME BEHAVIOUR CONSISTENCY CHECK


Declared income is compared with bank transaction behaviour from Step-3.


Example logic


monthly_income_expected = ITR_income / 12


Example


ITR declared income = 480000
Expected monthly income ≈ 40000


System compares:


* monthly deposits in bank statements
* platform earnings from Step-5
* business payments


If deposits >> expected income


System flags potential anomaly.


---


SERVER VERIFICATION (SIMULATED)


After validation, extracted data is sent to backend verification API.


Backend service simulates verification with the Income Tax Department filing system.


Endpoint

MongoDB Collection

Example response

Possible responses


* Valid
* Not Found
* Invalid Filing


---


AIS DATA CONSISTENCY (SIMULATED)


Backend performs simulated comparison with Annual Information Statement (AIS) data.


AIS contains financial activity such as:


* interest income
* investment records
* high value transactions


This verification layer helps detect inconsistencies between declared income and financial records.


Note: AIS comparison is simulated in the backend for demonstration.

SYSTEM DESIGN FIX IMPLEMENTATION PROMPT
(GIG WORKER FINTECH VERIFICATION PIPELINE)
Context
We are building a fintech verification system for gig workers using a hybrid architecture:
Validation Layer -> On-device processing
Verification Layer -> Server APIs (FastAPI simulation)
The onboarding pipeline contains 8 verification steps.
This document defines approved logical and technical fixes that must be implemented without changing the overall architecture.
Steps that remain unchanged must NOT be modified.
________________


STEP-1 -- BASIC PROFILE NORMALIZATION (IMPLEMENT FIX)
Issue
Names and addresses may appear in different formats across documents.
Later steps perform cross-step validation using these fields.
Example problem
Step-1 name:
Ravi Kumar
PAN card name:
RAVI KUMAR
Without normalization this may fail equality checks.
Required Fix
Before storing Step-1 profile data, perform normalization.
Normalization rules
Name
* convert to uppercase
* remove extra spaces
* remove special characters
Example
Input
" Ravi Kumar "
Normalized
"RAVI KUMAR"
Address normalization
* trim spaces
* convert to lowercase for comparison
Technical Implementation
Create a preprocessing function:
normalize_identity_fields()
Example
normalized_name = normalize(name)
Store normalized values in global profile object.
Example global profile
{
"name": "RAVI KUMAR",
"mobile": "9876543210",
"work_type": "Platform Worker"
}
________________


STEP-2 -- IDENTITY VERIFICATION FORMAT VALIDATION (IMPLEMENT FIX)
Issue
Current architecture may call verification APIs before validating input format.
This wastes API calls and increases attack surface.
Required Fix
All identifier inputs must pass format validation before server verification.
Correct validation order
User enters identifier
↓
Format validation (on-device)
↓
API verification
↓
Enable document upload
↓
OCR extraction
↓
Cross validation
Example
PAN validation
Regex rule
[A-Z]{5}[0-9]{4}[A-Z]
Example valid
ABCDE1234F
Example invalid
ABC1234
If regex fails -> block API request.
This rule applies to
PAN
Aadhaar
IFSC
Vehicle number
Scheme IDs
Insurance policy numbers
ITR acknowledgement number
________________


STEP-3 -- BANK STATEMENT PARSER LIMITATION (IMPLEMENT FIX)
Issue
Bank statements have different layouts.
A universal OCR parser may produce incorrect transaction extraction.
Current decision
Only support three banks initially.
Example supported banks
SBI
HDFC
ICICI
Required Fix
Implement bank-specific parsing modules.
Flow
Upload bank statement
↓
OCR detects bank name
↓
Load bank-specific parser
↓
Extract transactions
Example logic
if bank_name == "SBI":
use SBI_parser()
if bank_name == "HDFC":
use HDFC_parser()
if bank_name == "ICICI":
use ICICI_parser()
If bank not supported
Show message:
"Currently this bank statement format is not supported."
This reduces parsing errors during prototype stage.

Do NOT enforce owner name match.

STEP-8 -- ITR INCOME CONSISTENCY VALIDATION (IMPLEMENT FIX)
Issue
Gig workers often have irregular monthly income.
Strict comparison between bank deposits and ITR income could incorrectly flag valid users.
Required Fix
Use tolerance-based validation instead of strict equality.
Example
ITR declared income
480000
Expected monthly income
480000 / 12 = 40000
Allow tolerance range
+/-40%
Validation rule
lower_bound = expected_income x 0.6
upper_bound = expected_income x 1.4
Example
Valid bank monthly income
24000 -> 56000
If income outside this range
System flags anomaly but does NOT reject ITR.
Instead mark as:
income_consistency = "medium"
