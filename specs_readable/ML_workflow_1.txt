DOCUMENT PURPOSE
?????????????????????????????????????????????????????????????????????????????

This specification defines the complete machine learning pipeline for the
GigCredit scoring engine. It covers synthetic data generation, per-pillar
model training, SHAP computation, meta-learner training, model conversion,
on-device inference, confidence adjustment, score combination, and final
output format. Every model, every parameter, every data flow is defined.
An AI agent reading this document must be able to implement the entire
scoring engine without any external reference.

CRITICAL ARCHITECTURE NOTE:
  The scoring engine runs ENTIRELY ON THE USER'S FLUTTER DEVICE.
  FastAPI backend is used ONLY for:
    - Government API verification (Aadhaar, PAN, RC, Insurance, ITR)
    - MongoDB storage of final ScoreReport after scoring completes
    - LLM report generation (multilingual explanation)
  NO credit scoring computation happens on the FastAPI server.

?????????????????????????????????????????????????????????????????????????????
ARCHITECTURAL OVERVIEW
?????????????????????????????????????????????????????????????????????????????

THE SCORING ENGINE HAS TWO DISTINCT PHASES:

PHASE A « OFFLINE (runs ONCE on development machine, Python)
  1. Synthetic data generation (5000 profiles, 95 features each)
  2. Per-pillar model training (5 ML models: P1≈P4, P6)
  3. 5-fold cross-validation on all 5 models
  4. SHAP coefficient extraction from all 5 models
  5. Meta-learner training (4 Logistic Regression models, one per work type)
  6. Model conversion: Python ? ONNX ? TFLite ? INT8 ? AES-256 .enc
  7. Constants export to scoring_constants.dart

PHASE B « ONLINE (runs on user's Flutter device, every scoring, Dart)
  1. Feature engineering from VerifiedProfile (95 features)
  2. Per-pillar TFLite model inference (P1, P2, P3, P4, P6)
  3. P5 + P7 + P8 scorecard computation in Dart
  4. Confidence Engine
  5. Meta-Learner inference (Logistic Regression in Dart)
  6. SHAP Explanation Engine
  7. ScoreReport assembly
  8. POST ScoreReport to FastAPI ? MongoDB storage

PILLAR MODEL ASSIGNMENT:
  P1 Income Stability        : XGBoost Regressor     (13 features)
  P2 Payment Discipline      : CatBoost Classifier   (15 features)
  P3 Debt Management         : XGBoost Regressor     (9 features)
  P4 Savings Behaviour       : LightGBM Regressor    (12 features)
  P5 Work & Identity         : Dart Scorecard        (18 features « no model)
  P6 Financial Resilience    : RandomForest Regressor(11 features)
  P7 Social Accountability   : Dart Scorecard        (10 features « no model)
  P8 Tax Compliance          : Dart Scorecard        (7 features « no model)
  Meta-Learner               : Logistic Regression ﻁ 4 (one per work type)

PART 1 « SYNTHETIC DATA GENERATION
?????????????????????????????????????????????????????????????????????????????

WHY SYNTHETIC DATA:
  GigCredit has no real user data at launch. Synthetic data is generated
  to train the models. Profiles must be statistically realistic « reflecting
  actual Indian gig worker financial patterns « so models learn genuine
  relationships, not artificial rules.

?????????????????????????????????????????????????????????????????????????
1.1  PROFILE STRUCTURE
?????????????????????????????????????????????????????????????????????????

Every synthetic profile contains exactly 95 features and pillar labels.

PROFILE COMPONENTS:
  work_type     : one of [platform_worker, vendor, tradesperson, freelancer]
  features[95]  : all feature values (computed from simulated financials)
  final_label   : float 0.0≈1.0 (overall creditworthiness)
  p1_label      : float 0.0≈1.0 (income quality)
  p2_label      : float 0.0≈1.0 (payment discipline)
  p3_label      : float 0.0≈1.0 (debt management)
  p4_label      : float 0.0≈1.0 (savings quality)
  p6_label      : float 0.0≈1.0 (resilience quality)

NOTE: P5, P7, P8 are scorecard pillars « no training labels needed.
      Their output is computed deterministically in Dart at inference time.

1.3  STATISTICAL DISTRIBUTIONS PER FEATURE
?????????????????????????????????????????????????????????????????????????

avg_monthly_income_norm (P1, idx 0):
  Generated from LogNormal distribution. Parameters vary by work type + tier:
    Platform Excellent: ?=10.5, ?=0.4 ? mean ? ?36,000/month
    Platform Poor:      ?=9.2,  ?=0.5 ? mean ? ?9,900/month
    Vendor Excellent:   ?=10.2, ?=0.5 ? mean ? ?27,000/month
  LogNormal chosen: income cannot be negative, is right-skewed in
  real Indian populations.
  Normalised: raw_income ﻐ (ANCHOR ﻁ 3), clamp [0.0, 1.0]

income_stability_cv (P1, idx 1):
  Generated from Beta distribution.
  Excellent tier: Beta(?=8, ?=2) ? most values 0.05≈0.20 (stable)
  Poor tier:      Beta(?=2, ?=5) ? most values 0.40≈0.70 (unstable)
  cv = stdDev ﻐ mean of 6 monthly credits, inverted: score = 1 ? min(cv, 1)

Monthly income time series (6 months, used for income_growth_slope):
  Generated using AutoRegressive(1) model:
    month_t = ? ﻁ month_(t-1) + ?
    ? = 0.7 (moderate autocorrelation « income this month predicts next)
    ? = Gaussian noise (? scales with tier)
  This creates realistic month-to-month correlation, not independent noise.

emi_to_income_ratio (P3, idx 28):
  Generated from Beta distribution. Bounded 0.0≈0.95.
  Excellent: Beta(?=2, ?=8) ? most near 0.15≈0.25
  Very Poor: Beta(?=6, ?=3) ? most near 0.55≈0.80

face_similarity_score (P5, idx 50):
  Excellent+: mostly 0.82≈0.95
  All tiers:  floor at 0.70 (below = blocked at onboarding, never reaches model)

Binary insurance features (P6 idx 67≈73):
  Generated from Bernoulli distribution. P(active) varies by tier:
  health_insurance_active: P=0.72 for Excellent, P=0.18 for Very Poor
  eshram_active:           P=0.85 for all tiers (most gig workers register)

1.4  LABEL GENERATION
?????????????????????????????????????????????????????????????????????????

FINAL CREDITWORTHINESS LABEL:
  The label is NOT generated by copying feature rules. It is generated
  by a noisy weighted formula approximating true creditworthiness:

  RAW_LABEL =
    0.25 ﻁ income_level_normalised
  + 0.20 ﻁ (1 ? emi_to_income_ratio)
  + 0.15 ﻁ utility_ontime_ratio
  + 0.10 ﻁ avg_balance_to_income
  + 0.08 ﻁ (1 ? income_stability_cv)
  + 0.07 ﻁ insurance_coverage_composite
  + 0.05 ﻁ kyc_completeness
  + 0.05 ﻁ scheme_participation_score
  + 0.05 ﻁ work_tenure_norm

  NOISE INJECTION:
    Gaussian noise with ? = 0.06 added to RAW_LABEL

  TIER ANCHOR BLEND:
    30٪ of label pulled toward tier center value
    Prevents all profiles in same tier having identical labels

  FINAL_LABEL = clip(NOISY_LABEL ﻁ 0.7 + TIER_ANCHOR ﻁ 0.3, 0.0, 1.0)

  The noise + tier blend ensures the label cannot be reproduced by simply
  copying the formula. Models must discover the underlying weighted pattern.

PILLAR-SPECIFIC LABELS:

  p1_label (income quality):
    0.35 ﻁ avg_monthly_income_norm
  + 0.25 ﻁ income_stability_cv
  + 0.20 ﻁ income_growth_slope
  + 0.20 ﻁ (1 ? income_gap_months_norm)
  + Gaussian noise ? = 0.05

  p2_label (payment discipline):
    0.40 ﻁ utility_ontime_ratio
  + 0.30 ﻁ bank_transaction_matching_ratio
  + 0.20 ﻁ digital_payment_ratio
  + 0.10 ﻁ balance_never_negative_ratio
  + Gaussian noise ? = 0.05

  p3_label (debt management):
    0.50 ﻁ (1 ? emi_to_income_ratio)
  + 0.30 ﻁ emi_payment_regularity
  + 0.20 ﻁ loan_purpose_encoded
  + Gaussian noise ? = 0.04

  p4_label (savings quality):
    0.35 ﻁ avg_closing_balance_norm
  + 0.30 ﻁ savings_rate_norm
  + 0.20 ﻁ (1 ? zero_balance_months_penalty_inverted)
  + 0.15 ﻁ balance_growth_slope
  + Gaussian noise ? = 0.05

  p6_label (resilience):
    0.35 ﻁ insurance_coverage_composite
  + 0.30 ﻁ avg_balance_to_monthly_income
  + 0.20 ﻁ (itr_submitted_binary + gst_filing_consistency) ﻐ 2
  + 0.15 ﻁ life_sum_assured_norm
  + Gaussian noise ? = 0.05

  income_gap_months_norm for p1_label synthesis:
    count(months where monthly_credit < 20٪ of avg) ﻐ 6

  insurance_coverage_composite for p6_label synthesis:
    weighted avg:
      health_insurance_active ﻁ 0.40
    + life_insurance_active   ﻁ 0.35
    + vehicle_insurance_activeﻁ 0.25 (if vehicle worker, else 0)

PART 4 « CROSS-VALIDATION
?????????????????????????????????????????????????????????????????????????????

5-FOLD CV PROCESS:
  Split 5000 profiles into 5 equal folds of 1000 each.
  Fold assignment: stratified by tier AND work type.

  Round 1: train on folds 2+3+4+5, validate on fold 1
  Round 2: train on folds 1+3+4+5, validate on fold 2
  Round 3: train on folds 1+2+4+5, validate on fold 3
  Round 4: train on folds 1+2+3+5, validate on fold 4
  Round 5: train on folds 1+2+3+4, validate on fold 5

  Average the 5 validation scores for each model.

ACCEPTANCE CRITERIA:
  |CV_score ? single_val_score| < 0.015 per model
  If difference > 0.015: model is overfit ?
  Retrain with stronger regularisation (increase reg_alpha, reg_lambda,
  min_child_weight, or reduce n_estimators).

FINAL MODEL:
  After CV passes, retrain on ALL 5000 profiles using the
  validated hyperparameters. This final model is what gets
  converted to TFLite ? .enc and deployed in Flutter.

PART 5 « SHAP COEFFICIENT EXTRACTION
?????????????????????????????????????????????????????????????????????????????

TOOL: shap.TreeExplainer (from shap Python library)

PROCESS (for each of the 5 trained models):
  1. Select 200 random profiles from training set
  2. Run TreeExplainer on the trained model
  3. Compute SHAP values for all features across 200 profiles
  4. Average SHAP values per feature across the 200 profiles
  5. These are the shap_coefficients

ALSO COMPUTE: population mean of every feature across all 5000 profiles.

STORAGE:
  Written directly into scoring_constants.dart as constant arrays.

  For P1 (13 features):
    const List<double> shap_p1 = [0.142, -0.083, 0.071, ...];  // 13 values
    const List<double> mean_p1 = [0.48, 0.22, 0.51, ...];      // 13 values

  Same structure for P2 (15), P3 (9), P4 (12), P6 (11).

ON-DEVICE SHAP EXPLANATION (Dart):
  For each feature i in a pillar:
    impact[i] = shap_coeff[i] ﻁ (user_value[i] ? population_mean[i]) ﻁ 100

  Positive impact ? feature helped this user's score
  Negative impact ? feature hurt this user's score

  EXAMPLE:
    Feature: income_stability_cv (idx 1)
    shap_coeff      = ?0.142  (higher volatility = bad ? negative coeff)
    user_value      = 0.18   (user's cv « they are stable)
    population_mean = 0.35   (average user cv)
    impact = ?0.142 ﻁ (0.18 ? 0.35) ﻁ 100 = +2.41 pts (positive!)
             user more stable than average ? negative feature ? positive impact

SELECTION PER PILLAR:
  Compute impact for all features in pillar.
  Sort by absolute impact value descending.
  Select top 3 positive impacts.
  Select top 2 negative impacts.
  These become the explanation strings for that pillar.

EXPLANATION TEXT TEMPLATES (stored in scoring_constants.dart):
  income_stability_cv positive:
    "Stable income across all {N} months"
  utility_ontime_ratio positive:
    "{pct}٪ of utility bills paid on time"
  emi_to_income_ratio negative:
    "EMI obligations are high relative to income"
  health_insurance_active_binary negative:
    "No active health insurance « reduces financial safety buffer"

PART 6 « META-LEARNER TRAINING (4 Logistic Regression Models)
?????????????????????????????????????????????????????????????????????????????

WHY 4 SEPARATE META-LEARNERS:
  The 4 work types have fundamentally different income structures.
  A single meta-learner finds average weights suboptimal for every type.
  Platform worker scores are dominated by P1 income + P5 identity.
  Vendor scores rely more on P3 debt management.
  Training separate combiners lets each work type's pillar importance
  be learned empirically from data « not hard-coded.

?????????????????????????????????????????????????????????????????????????
6.1  BUILDING META-LEARNER TRAINING DATA
?????????????????????????????????????????????????????????????????????????

STEP 1:
  Pass all 5000 training profiles through all 5 trained models.
  Compute P5 scorecard values using Dart formula (in Python equivalent).
  Compute P7 scorecard values using P7 formula.
  Compute P8 scorecard values using P8 formula.
  Now every training profile has 8 pillar scores:
  P1_raw, P2_raw, P3_raw, P4_raw, P5_raw, P6_raw, P7_raw, P8_raw

STEP 2:
  Apply confidence engine to each profile's pillar scores.
  (Use the true data completeness for each synthetic profile.)
  Get 8 confidence-adjusted pillar scores:
  P1_adj, P2_adj, P3_adj, P4_adj, P5_adj, P6_adj, P7_adj, P8_adj

STEP 3:
  Build meta-learner input vector for each profile (20 values):
  [P1_adj, P2_adj, P3_adj, P4_adj, P5_adj, P6_adj, P7_adj, P8_adj,
   conf_P1, conf_P2, conf_P3, conf_P4, conf_P5, conf_P6, conf_P7, conf_P8,
   data_completeness, secondary_bank_present, itr_filed, gst_filed]

STEP 4:
  Meta-learner label = final_label binarised at 0.60
  (final_label ? 0.60 ? class 1 = acceptable credit)
  (final_label < 0.60 ? class 0 = credit risk)

?????????????????????????????????????????????????????????????????????????
6.2  TRAINING FOUR LR MODELS
?????????????????????????????????????????????????????????????????????????

SPLIT TRAINING DATA BY WORK TYPE:
  Platform Worker profiles : 2000 ? train Meta-Learner A
  Vendor profiles          : 1250 ? train Meta-Learner B
  Tradesperson profiles    : 1000 ? train Meta-Learner C
  Freelancer profiles      :  750 ? train Meta-Learner D

ALGORITHM    : Logistic Regression (sklearn LogisticRegression)
FEATURES IN  : 20 (meta-learner input vector)
LABEL        : binary creditworthiness (0 or 1)

HYPERPARAMETERS (same for all 4):
  C (inverse regularisation) : 1.0
  solver                     : lbfgs
  max_iter                   : 1000
  random_state               : 42

STANDARDISATION (MANDATORY):
  StandardScaler fit on training data for each work type.
  Scaler parameters (mean + std per feature) exported to constants.
  Both LR coefficients AND scaler parameters written to
  scoring_constants.dart.

WHAT LR LEARNS PER WORK TYPE:
  Meta-Learner A (Platform Worker):
    Discovers P1 coefficient largest « income + screenshot verification
    is the most reliable predictor for platform workers.

  Meta-Learner B (Vendor):
    Discovers P3 coefficient larger relative to P1 « vendor cash income
    makes P1 less reliable; debt repayment becomes proportionally more
    important.

  Meta-Learner C (Tradesperson):
    P5 (skill certification) carries more weight « NSDC certification
    is a strong work legitimacy signal for tradespeople.

  Meta-Learner D (Freelancer):
    P1 and P2 both high « invoice cross-verification + payment
    discipline together predict creditworthiness best.

EXPORTED PER WORK TYPE:
  LR coefficients vector  : 20 values
  LR intercept            : 1 value
  StandardScaler mean     : 20 values
  StandardScaler std      : 20 values
  Total: 61 numbers per work type ? 244 numbers total

META-LEARNER OUTPUT CONVERSION (executed in Dart):
  standardised_input = (input_vec ? scaler_mean) ﻐ scaler_std
  logit     = dot(standardised_input, lr_coefficients) + intercept
  probability = 1.0 ﻐ (1.0 + exp(?logit))
  final_score = round(probability ﻁ 600 + 300)
  Range: 300 (probability=0.0) to 900 (probability=1.0)

PART 7 « CONFIDENCE ENGINE
?????????????????????????????????????????????????????????????????????????????

PURPOSE:
  When data for a pillar is sparse (documents not submitted, fallbacks
  used), the pillar score should be pulled toward 0.50 (neutral).
  A high score with no supporting data is not credible.
  A low score with no data should not unjustly penalise the applicant.

CONFIDENCE COMPUTATION PER PILLAR:
  For each pillar P, define its data sources (documents + verified fields).
  confidence[P] = verified_sources_present ﻐ total_possible_sources

  PILLAR CONFIDENCE SOURCE WEIGHTS:
    P1: bank_primary(0.40) + bank_secondary(0.10) + upi(0.20)
        + platform_screenshots(0.20) + itr(0.10)
    P2: utility_bills(0.50) + bank(0.30) + upi(0.20)
    P3: bank_emi_detected(0.40) + emi_screenshot(0.30)
        + loan_schedule(0.30)
    P4: bank(0.50) + ppf(0.20) + upi(0.20) + rent(0.10)
    P5: work_proof_verified(0.50) + kyc_verified(0.30)
        + platform_api(0.20)
    P6: insurance_submitted(0.60) + bank_buffer(0.40)
    P7: eshram(0.30) + pmsym(0.25) + pmjjby(0.20)
        + mudra(0.15) + ppf(0.10)
    P8: itr_submitted(0.60) + gst_filed(0.40)

SCORE ADJUSTMENT FORMULA:
  NEUTRAL = 0.50
  adjusted_score[P] = raw_score[P] ﻁ confidence[P]
                    + NEUTRAL ﻁ (1.0 ? confidence[P])

  Full confidence (1.0) ? score unchanged
  Zero confidence (0.0) ? score = 0.50 exactly
  Partial (e.g. 0.60)   ? score pulled 40٪ toward 0.50

?????????????????????????????????????????????????????????????????????????????

  XGBoost Booster             (P1, P3)

  Using onnx-tf library.
  Required because TFLite converter only accepts TF models.

STEP 4: TF SavedModel ? TFLite
  TFLite converter removes training-only components:
    Training gradients, optimizer states, loss function graph.
  Only forward-pass computation remains.
  File size reduces dramatically.

STEP 5: INT8 Quantisation (during TFLite conversion)
  Float32 weights (4 bytes) ? INT8 (1 byte) = 75٪ size reduction.
  4ﻁ inference speed improvement via INT8 hardware acceleration.
  Accuracy loss: 0.5≈1.5٪ RMSE increase (acceptable for prototype).
  Calibration dataset: 100 representative training profiles.
  Used to compute INT8 scale factors accurately.

STEP 6: AES-256-CBC Encryption ? .enc file
  Encryption key generated and stored in Android Keystore / iOS Secure Enclave.
  Key never exists outside hardware secure storage.
  TFLite file encrypted to binary .enc file.

FILE 5: lib/scoring/confidence_engine.dart
  CLASS  : ConfidenceEngine
  METHOD: ConfidenceResult compute(Map rawScores, VerifiedProfile vp)
    Compute conf[P1..P8] from document presence flags in vp
    Compute adjusted[P] = raw[P]ﻁconf[P] + 0.50ﻁ(1?conf[P])
    Return {adjusted: Map, confidence: Map}

FILE 6: lib/scoring/meta_learner.dart
  CLASS  : MetaLearner
  METHOD: MetaResult infer(ConfidenceResult cr, VerifiedProfile vp)
    Select LR coefficients by work type from constants
    Build 20-value input vector
    Standardise using scaler mean + std from constants
    Compute logit = dot(scaled, coeff) + intercept
    Compute probability = sigmoid(logit)
    Compute final_score = round(probability ﻁ 600 + 300)
    Clamp to [300, 900]
    Assign grade and grade_label
    Return MetaResult {final_score, grade, probability}

FILE 7: lib/scoring/shap_engine.dart
  CLASS  : ShapEngine
  METHOD: ShapResult explain(ScoringInput si, Map pillarRaw)
    For each ML pillar (P1,P2,P3,P4,P6):
      Compute impact[i] = shapCoeff[i] ﻁ (value[i] ? mean[i]) ﻁ 100
      Sort by abs(impact), select top 3 positive + top 2 negative
      Fill text templates from scoring_constants.dart
    Select global top_reasons and improvement_tips
    Return ShapResult

FILE 8: lib/scoring/score_assembler.dart
  CLASS  : ScoreAssembler
  METHOD: ScoreReport assemble(all results above)
    Build complete ScoreReport object
    Compute display_score per pillar (adjusted ﻁ 100, rounded)
    Assemble llm_payload for LLM generation step
    Set generated_at to DateTime.now().toIso8601String()
    Return ScoreReport

FILE 9: lib/scoring/scoring_engine.dart
  CLASS  : ScoringEngine (Main Orchestrator)
  METHOD: Future<ScoreReport> score(VerifiedProfile vp)
    Step 1: Hard block checks
    Step 2: featureEngineer.compute(vp)
    Step 3: interpreterPool.runParallelInference(scoringInput)
    Step 4: scorecards.computeP5 + P7 + P8
    Step 5: Apply debt band cap to P3
    Step 6: confidenceEngine.compute(rawScores, vp)
    Step 7: metaLearner.infer(confidenceResult, vp)
    Step 8: shapEngine.explain(scoringInput, rawScores)
    Step 9: assembler.assemble(all above results)
    Step 10: POST to /api/v1/score/submit
    Step 11: POST llm_payload to /api/v1/report/generate
    Step 12: cleanup.purge()
    Return ScoreReport

    Zero out all Float32List arrays before freeing
