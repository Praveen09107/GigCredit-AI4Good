DOCUMENT PURPOSE
================================================================================

This document defines the COMPLETE revised machine learning scoring pipeline
for GigCredit. It incorporates all corrections and improvements identified
during the specification review process. Every change is justified.

An AI agent reading this document must be able to implement the entire
scoring engine without any external reference.

CRITICAL ARCHITECTURE NOTE:
  The scoring engine runs ENTIRELY ON THE USER'S FLUTTER DEVICE.
  FastAPI backend is used ONLY for:
    - Government API verification (Aadhaar, PAN, RC, Insurance, ITR)
    - MongoDB storage of final ScoreReport after scoring completes
    - LLM report generation (multilingual explanation)
  NO credit scoring computation happens on the FastAPI server.


================================================================================
WHAT CHANGED AND WHY
================================================================================

CHANGE 1 — DEPLOYMENT METHOD
  OLD: Python .pkl -> ONNX -> TF SavedModel -> TFLite -> INT8 -> AES-256 .enc
  NEW: Python model -> m2cgen export_to_dart() -> pure .dart files
  WHY: The old 5-step conversion pipeline had documented failure points.
       TF-DF requires Flex delegates (custom ops), not pure TFLite.
       CatBoost ONNX conversion has known issues.
       m2cgen transpiles trained models to pure Dart arithmetic code.
       Zero runtime dependencies. Zero conversion fragility.
       Model becomes compiled ARM code — more secure than encrypted files.

CHANGE 2 — ALGORITHM ASSIGNMENT
  OLD: XGBoost (P1,P3) + CatBoost (P2) + LightGBM (P4) + RandomForest (P6)
  NEW: XGBoost (P1,P2,P3,P4) + RandomForest (P6)
  WHY: On <15,000 heterogeneous tabular samples, differences between
       XGBoost/CatBoost/LightGBM are within random seed variation
       (McElfresh et al., 2023). CatBoost's ordered boosting only benefits
       categorical feature encoding — all GigCredit features are continuous
       [0,1] floats, so ordered boosting provides zero benefit.
       RandomForest kept for P6: 6 of 11 features are sparse binary flags
       where bootstrap aggregation genuinely outperforms sequential boosting.
       Unified XGBoost simplifies to ONE framework + ONE conversion path.

CHANGE 3 — SYNTHETIC DATA SCALE
  OLD: 5,000 profiles
  NEW: 15,000 profiles
  WHY: Free to generate, dramatically improves model stability. Previous
       per-cell count was 250 samples (5000/4 work types/5 tiers). New
       per-cell count is 750 samples — 3x more robust.

CHANGE 4 — HYPERPARAMETER TUNING
  OLD: Manually specified hyperparameters
  NEW: Optuna Bayesian optimization with 5-fold stratified CV
  WHY: Automated search finds better hyperparameters in fewer trials.
       100 trials x 5 folds x 5 models = ~2,500 training runs.
       On 15,000 samples this completes in ~15-30 minutes.

CHANGE 5 — META-LEARNER
  OLD: 4 separate Logistic Regression models (one per work type)
  NEW: 1 Logistic Regression with work-type interaction terms
  WHY: Freelancer meta-learner previously trained on only 750 samples
       (high variance). Single LR on 15,000 samples with interaction
       terms captures work-type-specific behavior with 6.7x more data.

CHANGE 6 — TREE CONSTRAINTS (MANDATORY FOR m2cgen)
  OLD: n_estimators=300, max_depth=6
  NEW: n_estimators=100-150, max_depth=4
  WHY: m2cgen GitHub Issue #168 — tree_method='hist' produces incorrect
       code output. MUST use tree_method='exact'.
       Large ensembles cause RecursionError during code generation.
       150 trees at depth 4 produces ~200-300KB Dart per model.
       300 trees at depth 6 produces ~2.6MB Dart per model.
       Accuracy impact minimal with Optuna tuning on 15,000 samples.

CHANGE 7 — SCORE COMPUTATION METHOD (CONTRADICTION RESOLVED)
  OLD: Two conflicting methods existed:
       - ML Workflow: Logistic Regression meta-learner
       - Report Pipeline: Simple weighted sum
  NEW: ONLY Logistic Regression meta-learner (weighted sum DELETED)
  WHY: These produce completely different results. Only one can be used.
       LR meta-learner is trainable and adapts to data. Weighted sum
       is a rigid manual formula. LR is strictly superior.

CHANGE 8 — EXECUTION STRATEGY (CONTRADICTION RESOLVED)
  OLD: Phase-3 said "sequential". ML Workflow said "parallel Isolates".
  NEW: Sequential execution, no Isolates needed.
  WHY: With m2cgen, inference is pure Dart arithmetic (~2-5ms per model).
       Total scoring time ~15-20ms. No need for Isolate overhead.
       Eliminates the contradiction between Phase-3 and ML Workflow.

REMOVED FROM SPEC:
  - All .enc encrypted model files (p1.enc, p2.enc, etc.)
  - Android Keystore model decryption logic
  - TFLite Interpreter pool (model_interpreter_pool.dart)
  - INT8 quantisation step
  - ONNX conversion step
  - Model decryption at app startup
  - TFLite runtime dependency (tflite_flutter plugin)
  - Weighted sum score computation from Report Pipeline spec

-------------------------
2.3 STATISTICAL DISTRIBUTIONS PER FEATURE
-------------------------

(Same distributions as original ML workflow spec — LogNormal for income,
 Beta for ratios, AR(1) for time series, Bernoulli for binary flags.
 Only the sample counts change. All distribution parameters unchanged.)

avg_monthly_income_norm (P1, idx 0):
  Generated from LogNormal distribution. Parameters vary by work type + tier:
    Platform Excellent: mu=10.5, sigma=0.4 -> mean ~36,000/month
    Platform Poor:      mu=9.2,  sigma=0.5 -> mean ~9,900/month
    Vendor Excellent:   mu=10.2, sigma=0.5 -> mean ~27,000/month
  LogNormal chosen: income cannot be negative, is right-skewed in
  real Indian populations.
  Normalised: raw_income / (ANCHOR * 3), clamp [0.0, 1.0]

income_stability_cv (P1, idx 1):
  Generated from Beta distribution.
  Excellent tier: Beta(a=8, b=2) -> most values 0.05-0.20 (stable)
  Poor tier:      Beta(a=2, b=5) -> most values 0.40-0.70 (unstable)
  cv = stdDev / mean of 6 monthly credits, inverted: score = 1 - min(cv, 1)

Monthly income time series (6 months, used for income_growth_slope):
  Generated using AutoRegressive(1) model:
    month_t = rho * month_(t-1) + epsilon
    rho = 0.7 (moderate autocorrelation)
    epsilon = Gaussian noise (sigma scales with tier)

emi_to_income_ratio (P3, idx 28):
  Generated from Beta distribution. Bounded 0.0-0.95.
  Excellent: Beta(a=2, b=8) -> most near 0.15-0.25
  Very Poor: Beta(a=6, b=3) -> most near 0.55-0.80

Binary insurance features (P6 idx 67-73):
  Generated from Bernoulli distribution. P(active) varies by tier:
  health_insurance_active: P=0.72 (Excellent), P=0.18 (Very Poor)

(All other feature distributions remain exactly as specified in
 the original ML workflow spec. No distribution changes.)

import optuna
from xgboost import XGBRegressor
from sklearn.ensemble import RandomForestRegressor
from sklearn.model_selection import cross_val_score, StratifiedKFold

def create_xgb_objective(X, y, strat_labels):
    def objective(trial):
        params = {
            'tree_method': 'exact',          # LOCKED — m2cgen requirement
            'random_state': 42,
            'objective': 'reg:squarederror',
            'n_estimators': trial.suggest_int('n_estimators', 80, 150),
            'max_depth': trial.suggest_int('max_depth', 3, 4),
            'learning_rate': trial.suggest_float('learning_rate', 0.01, 0.3, log=True),
            'gamma': trial.suggest_float('gamma', 0.0, 1.0),
            'min_child_weight': trial.suggest_int('min_child_weight', 5, 30),
            'subsample': trial.suggest_float('subsample', 0.6, 1.0),
            'colsample_bytree': trial.suggest_float('colsample_bytree', 0.5, 1.0),
            'reg_alpha': trial.suggest_float('reg_alpha', 0.0, 0.5),
            'reg_lambda': trial.suggest_float('reg_lambda', 0.0, 0.5),
        }
        model = XGBRegressor(**params)
        cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)
        scores = cross_val_score(model, X, y, cv=cv, scoring='neg_mean_squared_error')
        return scores.mean()
    return objective

PART 5 — SHAP EXPLAINABILITY

-------------------------
5.1 SHAP EXTRACTION (OFFLINE)
-------------------------

For each of the 5 ML models, extract SHAP values using TreeExplainer:

import shap
import numpy as np
import json

shap_lookup = {}

for pillar_name, model, X_data, feature_names in all_models:
    explainer = shap.TreeExplainer(model)
    shap_values = explainer.shap_values(X_data)

    pillar_lookup = {}
    for feat_idx, feat_name in enumerate(feature_names):
        # Create 10 percentile bins
        values = X_data[:, feat_idx]
        bins = np.percentile(values, np.arange(0, 101, 10)).tolist()

        # Compute average SHAP value per bin
        bin_shap = []
        for b in range(10):
            if b < 9:
                mask = (values >= bins[b]) & (values < bins[b + 1])
            else:
                mask = (values >= bins[b])
            if mask.sum() > 0:
                bin_shap.append(float(np.mean(shap_values[mask, feat_idx])))
            else:
                bin_shap.append(0.0)

        pillar_lookup[feat_name] = {
            'bins': bins,
            'shap_values': bin_shap
        }

    shap_lookup[pillar_name] = pillar_lookup

# Export (~8-12KB JSON file)
with open('shap_lookup.json', 'w') as f:
    json.dump(shap_lookup, f, indent=2)

-------------------------
5.2 ON-DEVICE SHAP LOOKUP (Dart)
-------------------------

On the device, feature importance is computed via simple bin lookup:

  double getFeatureImpact(String pillar, String featureName, double value) {
    var entry = shapLookup[pillar]![featureName]!;
    List<double> bins = entry['bins'];
    List<double> shaps = entry['shap_values'];

    // Find which bin the value falls into
    int binIdx = 0;
    for (int i = 1; i < bins.length; i++) {
      if (value >= bins[i]) binIdx = i;
    }
    if (binIdx >= shaps.length) binIdx = shaps.length - 1;

    return shaps[binIdx];
  }

This replaces the previous linear approximation:
  OLD: impact = coefficient * (value - mean)
  NEW: impact = lookup_bin(value) -> pre-computed SHAP value

The binned approach captures non-linear SHAP relationships that
the linear approximation cannot. It is more accurate for tree models
because tree models have inherently non-linear feature effects.

File size: ~8-12KB for all 5 models (negligible).
Computation: microseconds (simple array index lookup).

-------------------------
5.3 TOP FACTOR EXTRACTION
-------------------------

After computing SHAP impacts for all features:

  1. Collect all (feature_name, impact_value) pairs across all pillars
  2. Sort by absolute impact value (descending)
  3. Top 3 POSITIVE impacts = "Strengths"
  4. Top 3 NEGATIVE impacts = "Areas for Improvement"
  5. Include in ScoreReport for LLM explanation generation

NOTE: SHAP values are for EXPLANATION ONLY. They do NOT affect the
      credit score calculation. The score is computed by the meta-learner.

PART 6 — META-LEARNER
================================================================================

-------------------------
6.1 META-LEARNER DESIGN
-------------------------

ONE Logistic Regression model with work-type interaction terms.

INPUT FEATURES (20 total):
  [0..7]   : 8 pillar scores (P1, P2, P3, P4, P5, P6, P7, P8)
  [8..11]  : 4 work-type one-hot (is_platform, is_vendor, is_trades, is_freelancer)
  [12..19] : 8 interaction terms (P1*is_platform, P1*is_vendor, P2*is_platform, P2*is_vendor,
              P1*is_trades, P1*is_freelancer, P2*is_trades, P2*is_freelancer)

NOTE: Only P1 and P2 interaction terms are included (the two highest-weighted
      pillars). Adding interaction terms for all 8 pillars would create 32
      interaction features — too many for 15,000 samples. P1 (weight 0.25)
      and P2 (weight 0.20) capture 45% of the score, so their work-type
      interactions matter most.

WHY SINGLE LR INSTEAD OF 4 SEPARATE LRs:
  - Uses all 15,000 samples instead of splitting into 4 groups
  - Freelancer group goes from 750 -> 15,000 training samples
  - Work-type-specific behavior captured via interaction terms
  - Fully interpretable: each coefficient has clear meaning
  - More robust, lower variance estimates

-------------------------
6.2 META-LEARNER TRAINING
-------------------------

from sklearn.linear_model import LogisticRegression
import numpy as np

def build_meta_features(pillar_scores, work_type_onehot):
    """
    pillar_scores: array of shape (N, 8) — P1 through P8
    work_type_onehot: array of shape (N, 4)
    Returns: array of shape (N, 20)
    """
    # Base features: 8 pillars + 4 work type flags
    base = np.concatenate([pillar_scores, work_type_onehot], axis=1)

    # Interaction terms: P1 and P2 x each work type
    p1 = pillar_scores[:, 0:1]  # shape (N, 1)
    p2 = pillar_scores[:, 1:2]  # shape (N, 1)
    interactions = np.concatenate([
        p1 * work_type_onehot,  # P1 x 4 work types = 4 features
        p2 * work_type_onehot,  # P2 x 4 work types = 4 features
    ], axis=1)

    return np.concatenate([base, interactions], axis=1)

# Build training features
# pillar_scores_train = cross-validated predictions from 5 ML models + 3 scorecards
X_meta = build_meta_features(pillar_scores_train, work_type_onehot_train)
y_meta = final_labels_train  # binary: 1 if creditworthy, 0 if not

# Train
meta_model = LogisticRegression(C=1.0, max_iter=1000, random_state=42)
meta_model.fit(X_meta, y_meta)

# Extract coefficients and intercept
meta_coefficients = meta_model.coef_[0].tolist()  # 20 values
meta_intercept = float(meta_model.intercept_[0])

# Export to Dart constants
with open('meta_coefficients.json', 'w') as f:
    json.dump({
        'coefficients': meta_coefficients,
        'intercept': meta_intercept
    }, f)

IMPORTANT: To avoid data leakage, pillar scores for meta-learner training
must be generated via CROSS-VALIDATION (out-of-fold predictions), NOT by
running the final trained models on the same training data. Use the 5-fold
CV from Optuna — the out-of-fold predictions from the best trial.

-------------------------
6.3 SCORE COMPUTATION (FINAL — NO ALTERNATIVES)
-------------------------

This is the ONLY method for computing the final credit score.
The weighted sum method from the Report Pipeline spec is DELETED.

  STEP 1: Compute all 8 pillar scores (ML models + scorecards)
  STEP 2: Apply confidence adjustment per pillar
  STEP 3: Build meta-features (20 inputs)
  STEP 4: Compute logit = dot(meta_features, coefficients) + intercept
  STEP 5: Compute probability = 1 / (1 + exp(-logit))
  STEP 6: Compute final_score = round(probability * 600 + 300)

  Score range: 300-900 (matching CIBIL scale)
  300 = worst possible (probability = 0.0)
  900 = best possible (probability = 1.0)

DART IMPLEMENTATION:

  double computeFinalScore(List<double> pillarScores, int workTypeIndex) {
    // Build work type one-hot
    List<double> workTypeOH = [0.0, 0.0, 0.0, 0.0];
    workTypeOH[workTypeIndex] = 1.0;

    // Build meta features (20 total)
    List<double> metaInput = [
      ...pillarScores,                              // 8 pillar scores
      ...workTypeOH,                                // 4 work type flags
      pillarScores[0] * workTypeOH[0],              // P1 x platform
      pillarScores[0] * workTypeOH[1],              // P1 x vendor
      pillarScores[0] * workTypeOH[2],              // P1 x trades
      pillarScores[0] * workTypeOH[3],              // P1 x freelancer
      pillarScores[1] * workTypeOH[0],              // P2 x platform
      pillarScores[1] * workTypeOH[1],              // P2 x vendor
      pillarScores[1] * workTypeOH[2],              // P2 x trades
      pillarScores[1] * workTypeOH[3],              // P2 x freelancer
    ];

    // Dot product
    double logit = metaIntercept;
    for (int i = 0; i < 20; i++) {
      logit += metaInput[i] * metaCoefficients[i];
    }

    // Sigmoid
    double probability = 1.0 / (1.0 + exp(-logit));

    // Scale to 300-900
    return (probability * 600 + 300).roundToDouble();
  }

GRADE ASSIGNMENT:
  Score >= 800 : Grade S (Exceptional)
  Score >= 720 : Grade A (Excellent)
  Score >= 640 : Grade B (Good)
  Score >= 560 : Grade C (Average)
  Score >= 480 : Grade D (Below Average)
  Score <  480 : Grade E (Poor)

RISK BAND:
  Score >= 651 : Low Risk
  Score 451-650 : Medium Risk
  Score <= 450 : High Risk

7.3 EXPECTED OUTPUT FORMAT
-------------------------

Each generated .dart file contains a single function:

    double var0;
    if (input[2] >= 0.4532) {

This is pure Dart arithmetic. No ML library, no TFLite, no ONNX.
The function takes a List<double> of features and returns a double.

For XGBoost regressors, the output is the raw prediction.
For RandomForest regressors, the output is the averaged tree predictions.
Both produce values in approximately [0.0, 1.0] range (trained on [0,1] labels).

-------------------------
7.4 m2cgen CRITICAL REQUIREMENTS
-------------------------

REQUIREMENT 1 — tree_method='exact':
  XGBoost models MUST be trained with tree_method='exact'.
  m2cgen Issue #168: tree_method='hist' produces silently incorrect
  Dart output because hist uses approximate split thresholds that
  m2cgen does not replicate correctly.
  This is a MANDATORY training parameter, not optional.

REQUIREMENT 2 — sys.setrecursionlimit(50000):
  m2cgen traverses the tree structure recursively during code generation.
  150 trees x depth 4 = ~2,250 nodes. Default Python recursion limit
  is 1,000. Must be increased before calling export_to_dart().

REQUIREMENT 3 — No NaN inputs:
  m2cgen Issue #476: XGBoost's native NaN routing through trees is NOT
  replicated in generated code. All features MUST have explicit numeric
  fallback values. No NaN must ever reach the generated scorer function.
  The GigCredit feature engineering spec already enforces this via
  per-feature fallback values. Add a sanitization guard in Dart (see Part 9).

REQUIREMENT 4 — n_estimators <= 150, max_depth <= 4:
  Prevents excessive code generation:
    150 trees x depth 4 = ~200-300KB Dart source per model (manageable)
    300 trees x depth 6 = ~2.6MB Dart source per model (too large)
  Flutter's AOT compiler handles ~1-1.5MB total Dart source well.
  Beyond 5MB total may slow compilation and hit analyser limits.

  double adjustForConfidence(double rawScore, double confidence) {

9.6 DYNAMIC SCORING (HOW SCORE EVOLVES)
-------------------------

The score updates each time the user completes a new step:

  After Step-3 (bank statement):
    P1, P2, P3, P4 have real data -> confidence high
    P5-P8 have partial/no data -> confidence low, scores near 0.50
    -> Provisional score (e.g., 520)

  After Step-5 (work documents):
    P5 now has real data -> confidence increases
    -> Score improves (e.g., 560)

  After Step-7 (insurance):
    P6 now has real data -> confidence increases
    -> Score improves (e.g., 625)

  Each step completion triggers re-inference (~15-20ms).
  The user sees their score grow in real time.

  This dynamic behavior comes from the Confidence Engine, NOT from
  the model architecture. It works identically with any model type.

PART 14 — SECURITY CONSIDERATIONS (REVISED)
================================================================================

WITH m2cgen DEPLOYMENT:

  MODEL SECURITY:
    - Models are compiled into the app binary as Dart code
    - Flutter AOT compiler transforms Dart -> ARM machine instructions
    - Tree logic becomes indistinguishable from other app code
    - Cannot be extracted as a model file (unlike .tflite/.enc)
    - R8/ProGuard obfuscation further protects the compiled output
    - This is MORE secure than encrypted model files, which are
      vulnerable to memory-dump attacks during decryption

  WHAT IS NO LONGER NEEDED:
    - Android Keystore for model encryption keys
    - AES-256-CBC model encryption/decryption
    - Runtime model integrity verification
    - Anti-tampering checks on model files

  WHAT IS STILL NEEDED:
    - Android Keystore for session data encryption (verified_profile)
    - Root/debugger/hook detection (protect scoring logic in memory)
    - App signature verification (prevent repackaging)
    - Secure cleanup of user financial data after report generation
    - All security measures for user data remain UNCHANGED
