GIGCREDIT — USER INPUT COLLECTION SPECIFICATION
Complete Onboarding Workflow | Steps 1 through 8
Version 2.0 | For Agent Implementation
DOCUMENT PURPOSE
This specification defines every user input collected during the GigCredit onboarding process. It is written for an implementation agent and covers the exact inputs required in each step, the format and UI method for collecting each input, whether each input is mandatory or optional, the total count of inputs per step, and how Step 5 dynamically reconfigures itself based on the gig worker category selected in Step 1. No validation logic is covered in this document.


SYSTEM OVERVIEW
text

Steps With Dynamic Content: Step 5 (changes per work type)
Steps With Conditional Content: Step 4 (rent), Step 7 (vehicle)

  1. Platform Worker
  2. Vendor / Seller

  4. Freelancer

  → Entire content of Step 5

STEP 2 — IDENTITY VERIFICATION
Purpose
Verify the user's government-issued identity documents and confirm that the person physically present matches the identity on file. This step produces the KYC verification status used in P5 identity scoring.


How This Step Is Presented
Three separate upload cards, each clearly labelled, with camera/gallery options for document images and an in-app camera capture trigger for the selfie. Each card shows a guide overlay or sample image illustrating the correct way to photograph the document.

1        Aadhaar Number        Numeric text field, segmented 4-4-4        12-digit number        1 entry        ✅ Yes

3        Aadhaar Card — Back Side        Image upload button        JPG or PNG, camera or gallery        1 image        ✅ Yes
text
Why both sides are required:
  Front: name, date of birth, Aadhaar number, photo
  Back:  full address including state and PIN code
  Back address is needed for cross-matching against
  utility bill addresses and confirming state of residence
PAN Card
#        Field Name        UI Element        Input Format        Count        Mandatory
4        PAN Number        Alphanumeric text field        10 characters (ABCDE1234F format)        1 entry        ✅ Yes
5        PAN Card Photo        Image upload button        JPG or PNG, camera or gallery        1 image        ✅ Yes
Live Selfie
#        Field Name        UI Element        Input Format        Count        Mandatory
6        Live Selfie        In-app camera trigger button        Camera capture only — no gallery upload permitted        1 photo        ✅ Yes
text
Why camera-only for selfie:
  Gallery uploads could be a photo of someone else's photo.
  Camera-only with liveness instructions ensures the person
  submitting is physically present at time of capture.
  The captured selfie is compared against the Aadhaar front

What Is Extracted After Step 2 Submission
text
Aadhaar Front OCR  → Name, date of birth, Aadhaar number
Aadhaar Back OCR   → Full address, state, PIN code

                     and ITR filing status (used in Steps 6 and 8)

                     face against Aadhaar front photo

How This Step Is Presented
Three mandatory bill sections displayed as expandable accordion panels — Electricity, LPG/Gas, and Mobile. Each panel contains a 6-slot upload grid labelled by month. Below the mandatory panels sit two optional panels for additional utility types and one conditional panel for rent.

STEP 5 — GIG WORK PROOF (DYNAMICALLY CONFIGURED)
Purpose
Verify the user's occupation, work history, and work-specific income sources. This step is entirely reconfigured based on the Work Type selected in Step 1. The screen title, instructions, upload cards, and all fields are unique to each work type.


How Dynamic Configuration Works
text
When user reaches Step 5, the system reads the
Work Type stored from Step 1 and renders one of
four completely different screen layouts:


  Work Type = Platform Worker    → Render Layout 5A
  Work Type = Vendor / Seller    → Render Layout 5B
  Work Type = Tradesperson       → Render Layout 5C
  Work Type = Freelancer         → Render Layout 5D


No work type sees another work type's inputs.
Features not applicable to a work type are set to
structural zero during feature engineering —
they are absent by design, not by missing data.

STEP 8 — ITR AND GST RECORDS
Purpose
Collect formal income declaration and tax compliance records. ITR and GST documents independently verify the income reported in the bank statement and declared in Step 1. These are the strongest formal income signals in the system and improve confidence scores across P1, P4, P6, and the meta-learner.


How This Step Is Presented
Two separate sections on one screen — ITR section on top and GST section below. Both sections are optional and collapsed by default. A clear message at the top of the screen explains the score benefit of submitting. GST section is visually emphasised for Vendor and Freelancer work types.


Step-level message shown to user:
"ITR and GST records are the strongest proof of income. Submitting them can improve your score by up to 22 points across multiple pillars."
