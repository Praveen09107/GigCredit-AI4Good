MASTER SYSTEM DESIGN PROMPT — COMPLETE ON-DEVICE VERIFICATION, VALIDATION, SECURITY, AND CREDIT SCORING PIPELINE


SYSTEM CONTEXT


Design a privacy-first mobile verification system called GigCredit that evaluates the financial credibility of gig workers who typically lack traditional credit scores.


The system verifies identity, validates financial behaviour, detects document manipulation, and generates a trustworthy alternative credit profile.


The architecture must combine:


• On-device OCR extraction and validation
• AI image authenticity detection
• Multi-layer document verification
• Backend verification APIs
• MongoDB verification datasets
• Cross-step identity validation
• Structured financial transaction analysis
• Secure AI model execution


The core rule of the system:


ONLY VERIFIED DATA PROCEEDS TO SCORING.


Any data failing validation is rejected and excluded from the scoring pipeline.


---


SYSTEM ARCHITECTURE OVERVIEW


The system consists of two major components.


1. On-Device Processing Layer
2. Backend Verification Layer


Sensitive documents remain on the device and are processed locally.


Backend APIs simulate official verification services.

---


MODEL ENCRYPTION


During development the evaluation model is encrypted.


Example workflow


evaluation_model.tflite
↓
AES encryption
↓
evaluation_model.enc


Only the encrypted file is packaged inside the mobile application.


Example


assets/evaluation_model.enc


If the application is reverse engineered the attacker only sees encrypted data.


---


ANDROID KEYSTORE


The encryption key used for model decryption must be securely stored.


The system uses Android Keystore.


Key generation process


First app launch
↓
Generate encryption key
↓
Store key in Android Keystore


Key properties


• Key stored in secure hardware
• Key cannot be exported
• Key cannot be accessed by other applications


The application can only request cryptographic operations using the key.


---


RUNTIME MODEL DECRYPTION


The encrypted model is decrypted only during runtime.


Example runtime process


evaluation_model.enc
↓
Retrieve key from Android Keystore
↓
Decrypt model in memory
↓
Load TensorFlow Lite interpreter


The decrypted model exists only in RAM.


It is never written to disk.


---


CODE OBFUSCATION


Application code is obfuscated during build.


Purpose


• Protect sensitive logic
• Prevent reverse engineering
• Hide model loading functions


Example transformation


loadEvaluationModel() → a.b()


---


DEBUGGER DETECTION


The system checks whether a debugger is attached.


If debugger detected


Terminate execution or disable AI model inference.


---


ROOT DETECTION


The system checks whether the device is rooted.


Indicators include


• Superuser binaries
• Root management apps
• Modified system partitions


If root detected


Disable document verification modules.


---


ANTI HOOK DETECTION


The system checks for runtime hooking frameworks.


Example


Frida detection.


If detected


Block model execution.


---


SECURE DATA CLEANUP


After verification completes


• Uploaded images deleted
• OCR text cleared
• Temporary files removed
• Model unloaded from memory


This prevents sensitive data leakage.
