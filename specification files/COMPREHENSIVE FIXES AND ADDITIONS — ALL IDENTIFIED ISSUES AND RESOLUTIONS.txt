AFFECTED FILES: Phase-3 Section-21, ML Workflow

  RESOLUTION:

-------------------------
FIX-A6: Face Verification "Review" State Undefined
AFFECTED FILES: Phase-3 Section-7
STATUS: RESOLVED
-------------------------

  CONFLICT:
    Spec defines three outcomes: >= 0.75 verified, 0.60-0.75 review, < 0.60 reject
    "Review" has no operational definition — no one reviews it.

  RESOLUTION:
    >= 0.75         : PASS (verified)
    0.60 - 0.75     : AUTO-RETRY with guidance
      -> Show: "Face match unclear. Please retake selfie with better lighting."
      -> On-screen guidance: face camera directly, good lighting, remove accessories
      -> Maximum 2 retries
      -> If all retries score 0.60-0.75: ACCEPT with reduced confidence
         face_match_confidence = 0.65
         face_match_quality = "LOW"
         This reduces P5 pillar confidence
    < 0.60          : REJECT
      -> "Face verification failed. Identity could not be confirmed."
      -> Block Step-2 completion
      -> User must re-upload documents

-------------------------
FIX-A7: Confidence Engine Collapse at Zero Data
AFFECTED FILES: ML Workflow, Confidence Engine
STATUS: RESOLVED
-------------------------

  PROBLEM:
    Formula: adjusted = raw * confidence + 0.50 * (1 - confidence)
    If confidence = 0.0, ALL pillar scores become 0.50
    User with zero data gets a "neutral" score instead of rejection

  RESOLUTION:
    MINIMUM DATA REQUIREMENTS (must have ALL):
      - Step-1 completed (basic profile)
      - Step-2 completed (identity verified)
      - Step-3 completed (at least 1 bank statement with >= 30 transactions)

    IF MINIMUM NOT MET:
      -> Do NOT compute score
      -> Show: "Insufficient data for credit assessment. Please complete Steps 1-3."

    CONFIDENCE FLOOR:
      If confidence < 0.30 for any pillar:
        -> Set pillar score to 0.50 (neutral, excluded from meaningful scoring)
        -> Report shows "Not enough data" for that pillar
        -> Meta-learner still receives 0.50 for this pillar

      1. Save verified_profile to local storage (encrypted)

      4. Resume from last incomplete step

    Session data valid for 24 hours.
    After 24 hours: delete session data, restart from Step-1.
    Rationale: prevents stale financial data from being scored.

  ENCRYPTION:
    Same AES key from Android Keystore used for session data.
    This key is NOT used for model encryption (models are now Dart code).
    Keystore key management remains for user data protection only.

FIX-B4: API Authentication
AFFECTED FILES: Master Backend Specification
-------------------------

  SPECIFICATION:

  FOR HACKATHON PROTOTYPE: API Key + Device Fingerprint

  FLOW:
    1. App ships with embedded API key (hardcoded constant, obfuscated)
    2. On first launch, generate device_id (UUID v4, stored in Keystore)
    3. Every API request includes:
       Headers:
         X-API-Key: <api_key>
         X-Device-ID: <device_uuid>
         X-Timestamp: <unix_ms>
         X-Signature: HMAC-SHA256(api_key, device_id + timestamp + body_hash)
    4. Backend validates:
       - API key is valid
       - Signature matches
       - Timestamp within 5-minute window (replay prevention)

  RATE LIMITING:
    Per device: 60 requests/minute
    Per endpoint: 10 requests/minute
    Burst: 5 requests/second
    Exceeded: return HTTP 429 with Retry-After header

  WHY NOT JWT:
    - App has no user accounts in current spec
    - JWT requires registration, token refresh, logout handling
    - API key + HMAC is simpler, stateless, sufficient for prototype
    - If user accounts added later, upgrade to JWT

  BACKEND IMPLEMENTATION:
    FastAPI middleware validates X-API-Key and X-Signature on every request.

-------------------------
FIX-C1: Transaction Tagging Engine (Enhanced)
AFFECTED FILES: Phase-3 Section-12, Feature Engineering
-------------------------

  CURRENT PROBLEM:
    Transaction tagging uses simple keyword matching.
    Indian bank statements have wildly inconsistent formats
    (UPI/NEFT/IMPS/RTGS all look different).

  ENHANCED TAGGING ENGINE (4 layers):

  LAYER 1 — REGEX KEYWORD MATCHING (baseline):
    GIG PLATFORM KEYWORDS:
      ["SWIGGY", "ZOMATO", "OLA", "UBER", "RAPIDO", "DUNZO", "PORTER",
       "URBAN COMPANY", "URBANCLAP", "FLIPKART", "AMAZON FLEX", "BLINKIT",
       "ZEPTO", "BIGBASKET", "INSTAMART", "MEESHO", "UPWORK", "FIVERR",
       "FREELANCER"]
    EMI KEYWORDS: ["EMI", "LOAN EMI", "NACH EMI", "FINANCE", "BAJAJ", "CARD EMI", "BNPL"]
    UTILITY KEYWORDS: ["TNEB", "BESCOM", "MSEDCL", "BSES", "TATA POWER",
       "AIRTEL", "JIO", "VI ", "BSNL", "WATER BOARD", "GAS"]

    Tags: GIG_INCOME, SALARY, EMI_DEBIT, UTILITY_PAYMENT, SAVINGS,
          P2P_TRANSFER, CASH_WITHDRAWAL, BANK_CHARGES

  LAYER 2 — PATTERN MATCHING (new):
    UPI: "UPI/P2P/{sender}/{receiver}/{id}" -> extract sender/receiver
    NEFT: "NEFT/{bank}/{name}/{ref}" -> extract by position
    IMPS: "IMPS/{ref}/{name}" -> extract name
    NACH: "NACH/{mandate}/{ref}" -> always recurring = EMI or SUBSCRIPTION

  LAYER 3 — AMOUNT-BASED HEURISTICS (new):
    Credits INR 200-2,000 with frequency > 10/month -> likely GIG_INCOME
    Identical debit amounts monthly -> likely EMI or SUBSCRIPTION
    Large single credits > INR 20,000 -> likely SALARY or LOAN_DISBURSEMENT
    Small debits INR 1-100 -> likely BANK_CHARGES

  LAYER 4 — UNTAGGED FALLBACK (new):
    Transactions matching no pattern -> tag as "UNCATEGORIZED"
    UNCATEGORIZED still counts toward:
      - Total transaction count
      - Average balance computation
      - Cash flow volatility
    Does NOT count toward specific pillar features like gig_income_count
